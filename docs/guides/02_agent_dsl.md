# The Agent DSL

Complete reference for every DSL method available when defining a Spurline agent. All DSL methods are class-level declarations that register configuration at class load time. They never execute behavior.

---

## Class Hierarchy

```
Spurline::Base             # Framework internals. Includes all DSL modules.
  Spurline::Agent          # Public API. Developers inherit from this.
    ApplicationAgent       # Generated by `spur new`. Project-wide defaults.
      YourAgent            # Your agent classes.
```

`Spurline::Base` includes the six DSL modules and provides shared registries (tool registry, adapter registry, session store). `Spurline::Agent` adds `#run`, `#chat`, session management, and lifecycle hooks. You never subclass `Base` directly.

---

## DSL Modules at a Glance

| Module           | Method         | Purpose                          |
|------------------|----------------|----------------------------------|
| `DSL::Model`     | `use_model`    | Select the LLM adapter           |
| `DSL::Persona`   | `persona`      | Define named system prompts      |
| `DSL::Tools`     | `tools`        | Declare available tools          |
| `DSL::Memory`    | `memory`       | Configure memory stores          |
| `DSL::Guardrails`| `guardrails`   | Set security and safety limits   |
| `DSL::Hooks`     | `on_*` methods | Register lifecycle callbacks     |

Each module lives in `lib/spurline/dsl/`.

---

## use_model

Selects the LLM adapter. Takes a symbol name and optional keyword arguments.

```ruby
use_model :claude_sonnet
use_model :claude_sonnet, max_tokens: 4096, temperature: 0.7
```

Stores `{ name: :claude_sonnet, max_tokens: 4096, temperature: 0.7 }` in `@model_config`. At instantiation, the agent resolves the adapter by looking up `config[:name]` in the adapter registry. There is no load-time validation -- invalid adapter names raise `AdapterNotFoundError` at instantiation.

**Inheritance:** child overrides entirely, or inherits if not set. Access via `MyAgent.model_config`.

---

## persona

Defines a named system prompt. The name defaults to `:default` if omitted.

```ruby
persona(:default) do
  system_prompt "You are a research assistant."
end
```

The block receives a `PersonaConfig` object evaluated with `instance_eval`:

| Method                       | Default | Description                                       |
|------------------------------|---------|---------------------------------------------------|
| `system_prompt(text)`        | `""`    | The system prompt sent to the LLM                 |
| `inject_date(bool)`          | `false` | Adds `Current date: YYYY-MM-DD` as a system supplement at assembly time |
| `inject_user_context(bool)`  | `false` | Adds `Current user: ...` when `session.user` is present |
| `inject_agent_context(bool)` | `false` | Adds agent metadata (class name, available tools) as a system supplement |

Supplements are injected dynamically on each run/chat turn during context assembly (not at class load time). This keeps date/user/agent fields current while the compiled persona remains immutable.

### Multiple Personas

Define several and select at instantiation:

```ruby
class SupportAgent < ApplicationAgent
  persona(:friendly) do
    system_prompt "You are a warm, encouraging support agent."
  end

  persona(:formal) do
    system_prompt "You are a professional support representative."
  end
end

agent = SupportAgent.new(persona: :formal)
```

If no `persona:` keyword is passed to `.new`, the agent resolves `:default`.

**Inheritance:** personas merge. A child inherits all parent personas. A same-name persona in the child wins. Access via `MyAgent.persona_configs`.

## tools

Declares which registered tools this agent can invoke. Accepts bare symbols and keyword arguments for per-tool config:

```ruby
tools :web_search,
      file_delete: { requires_confirmation: true, timeout: 30 },
      database_query: { timeout: 60 }
```

This registers three tools: `:web_search` (default config), `:file_delete`, and `:database_query`.

### Per-Tool Secret Overrides

Use `secrets:` inside tool config to control how declared tool secrets are resolved for this agent:

```ruby
tools send_email: {
  secrets: {
    sendgrid_api_key: :transactional_sendgrid_key
  }
}
```

In this example, a tool-declared `secret :sendgrid_api_key` resolves from:
`Spurline.credentials["transactional_sendgrid_key"]` before default lookup.

Override values can be:

- `Symbol` or `String`: credential key mapping
- `Proc`/`Method`: computed value at runtime
- Any other literal: used directly

### Accessing Configuration

```ruby
MyAgent.tool_config
# => { names: [:web_search, :file_delete, :database_query],
#      configs: { file_delete: { requires_confirmation: true, timeout: 30 },
#                 database_query: { timeout: 60 } } }

MyAgent.tool_config_for(:file_delete)
# => { requires_confirmation: true, timeout: 30 }
```

### Registration

The `tools` DSL declares names. The tools themselves must be registered separately:

```ruby
MyAgent.tool_registry.register(:calculator, Calculator)
```

A declared name missing from the registry raises `Spurline::ToolNotFoundError` at runtime.

**Inheritance:** names and configs merge. Child tools are added to the parent set (deduplicated). Child configs override parent configs for the same tool name.

## memory

Configures memory stores.

```ruby
memory :short_term, window: 30
memory :long_term,
       adapter: :postgres,
       connection_string: ENV.fetch("DATABASE_URL"),
       embedding_model: :openai
```

- `:short_term` uses a sliding window (`window:` default is 20).
- `:long_term` supports `adapter: :postgres` and `embedding_model: :openai`.
- `:long_term` can also accept custom adapter/embedder objects that respond to the expected interface.

**Inheritance:** configs merge per memory type. Access via `MyAgent.memory_config`.

---

## guardrails

Configures security and safety constraints. The block receives a `GuardrailConfig` object:

```ruby
guardrails do
  injection_filter :strict
  pii_filter :redact
  max_tool_calls 5
  max_turns 20
  audit_max_entries 5000
  denied_domains ["evil.example.com"]
  audit :full
end
```

### Available Settings

| Method              | Valid Values                              | Default   |
|---------------------|-------------------------------------------|-----------|
| `injection_filter`  | `:strict`, `:moderate`, `:permissive`     | `:strict` |
| `pii_filter`        | `:redact`, `:block`, `:warn`, `:off`      | `:off`    |
| `max_tool_calls`    | Any positive integer                      | `10`      |
| `max_turns`         | Any positive integer                      | `50`      |
| `audit_max_entries` | Any positive integer                      | `nil`     |
| `denied_domains`    | Array of domain strings                   | `[]`      |
| `audit`             | `:full`, `:errors_only`, `:off`           | `:full`   |

### Load-Time Validation

Every setting is validated the moment the DSL block executes. Invalid values raise `Spurline::ConfigurationError` at class load time:

```ruby
guardrails do
  injection_filter :nonexistent
  # => ConfigurationError: Invalid guardrail value for injection_filter: :nonexistent.
  #    Must be one of: :strict, :moderate, :permissive.
end
```

Integer settings reject non-positive values and non-integers:

```ruby
guardrails do
  max_tool_calls 0    # => ConfigurationError: Must be a positive integer.
  max_tool_calls "10" # => ConfigurationError: Must be a positive integer.
  audit_max_entries 0 # => ConfigurationError: Must be a positive integer.
end
```

**Inheritance:** if a subclass defines a `guardrails` block, it replaces the parent config entirely. No per-field merge. Access via `MyAgent.guardrail_config` (returns `GuardrailConfig`; call `.to_h` or `.settings` for the hash).

## Lifecycle Hooks

Register callbacks for key moments in the agent lifecycle. Each hook takes a block. Multiple hooks per type are allowed and fire in registration order.

```ruby
on_start  { |session| puts "Started: #{session.id}" }
on_finish { |session| puts "Finished: #{session.id}" }
on_error  { |error| Bugsnag.notify(error) }
```

### Available Hooks

| Hook             | Block Argument | When It Fires                     |
|------------------|----------------|-----------------------------------|
| `on_start`       | `session`      | After `initialize` completes      |
| `on_turn_start`  | `session`      | Before each LLM call in the loop  |
| `on_tool_call`   | `tool_call`    | Before a tool is executed         |
| `on_turn_end`    | `session`      | After each LLM call completes     |
| `on_finish`      | `session`      | After `#run` or `#chat` completes |
| `on_error`       | `error`        | When an `AgentError` is raised    |

Register multiple callbacks for the same type by calling it more than once:

```ruby
on_error { |e| Logger.error(e.message) }
on_error { |e| Metrics.increment("agent.errors") }
```

**Inheritance:** hooks accumulate. A child's hooks are appended after the parent's hooks for each type. They never replace the parent's hooks.

---

## Inheritance Summary

| DSL Method   | Strategy                                                      |
|--------------|---------------------------------------------------------------|
| `use_model`  | Override. Child replaces parent entirely, or inherits if unset |
| `persona`    | Merge. Same-name persona in child wins; others inherited       |
| `tools`      | Merge. Names deduplicated; child configs override per-tool     |
| `memory`     | Merge. Child overrides same-type config; others inherited      |
| `guardrails` | Replace. Child replaces entire config, or inherits if unset    |
| Hooks        | Accumulate. Child hooks appended after parent hooks            |

Registries (`tool_registry`, `adapter_registry`) and `session_store` are shared via the `inherited` hook in `Spurline::Base`. All subclasses in a process share the same registry instances.

## Complete Example

```ruby
class ApplicationAgent < Spurline::Agent
  use_model :claude_sonnet

  guardrails do
    injection_filter :strict
    pii_filter :off
    max_tool_calls 10
    max_turns 50
    audit_max_entries 5000
    audit :full
  end

  memory :short_term, window: 20
  # memory :long_term, adapter: :postgres, connection_string: "...", embedding_model: :openai
  on_error { |e| ErrorTracker.report(e) }
end

class ResearchAgent < ApplicationAgent
  persona(:default) do
    system_prompt "You are a thorough research assistant. Cite sources."
  end

  persona(:brief) do
    system_prompt "You are a concise research assistant. Be terse."
  end

  tools :web_search, :calculator,
        file_delete: { requires_confirmation: true }

  guardrails do
    injection_filter :strict
    pii_filter :redact
    max_tool_calls 15
    max_turns 30
    denied_domains ["malware.example.com"]
    audit :full
  end

  memory :short_term, window: 40

  on_start     { |s| puts "Research session: #{s.id}" }
  on_tool_call { |tc| puts "Calling: #{tc}" }
end
```

```ruby
agent = ResearchAgent.new(user: "analyst", persona: :brief)
agent.run("Find recent papers on transformer architectures") do |chunk|
  print chunk.text if chunk.text?
end
```

---

## What the DSL Does Not Do

- DSL methods never make LLM calls. All execution happens in `#run` and `#chat`.
- DSL methods never validate tool existence at load time. `ToolNotFoundError` fires at runtime.
- DSL methods do not configure the context pipeline directly. The pipeline reads guardrail settings at instantiation.
- `use_model` is not validated at load time. `AdapterNotFoundError` fires at instantiation.
