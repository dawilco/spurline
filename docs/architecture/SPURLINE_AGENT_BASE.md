# Spurline — Priority 1: Agent Base Class
## Core DSL, Lifecycle, and LLM Call Loop

---

## Overview

The Agent base class is the heart of Spurline. Everything else in the framework connects through it. It must be designed with extreme care because it defines the contract every agent author works against — changing it later breaks every downstream project.

Three responsibilities live here:

1. **The DSL** — the declarative interface developers write
2. **The Lifecycle** — how an agent is initialized, run, and torn down
3. **The LLM Call Loop** — the think → act → observe cycle that drives agent behavior

---

## Class Hierarchy

```
Spurline::Base                    # Framework internals, never touched by developers
  └── Spurline::Agent             # The framework base class — ships with spurline-core
        └── ApplicationAgent      # The developer's shared base — generated by `spur new`
              └── ResearchAgent   # A specific agent in the application
              └── SchedulingAgent
```

`Spurline::Base` handles low-level wiring — registry access, adapter loading, pipeline setup. Developers never interact with it directly. `Spurline::Agent` is the public API. `ApplicationAgent` is what `spur new` generates, mirroring Rails' `ApplicationController` pattern.

## The DSL

The DSL is purely declarative. When the class is loaded, DSL calls register configuration on the class — they do not execute behavior. Behavior only happens when `#run` is called.

### Model Configuration

```ruby
class ApplicationAgent < Spurline::Agent
  use_model :claude_sonnet

  # or with explicit options
  use_model :claude_sonnet, temperature: 0.7, max_tokens: 4096
end
```

`use_model` maps to an adapter. The symbol is resolved against registered adapters at boot time — not at call time — so misconfiguration fails fast on startup, not mid-conversation.

### Persona

The persona defines the system prompt and context injections. It is versioned and immutable at runtime — it cannot be changed by user input or tool results.

```ruby
persona :assistant do
  system_prompt "You are a helpful assistant for Acme Corp."
  inject_date          true   # appends current date to system prompt
  inject_user_context  true   # appends current_user metadata if available
  inject_agent_context true   # appends agent name, version, capabilities
end
```

The persona block is evaluated at class load and compiled into a `Content` object with trust level `:system`. It never passes through the injection scanner — it is trusted by definition. Attempting to modify it at runtime raises `PersonaFrozenError`.

Multiple personas can be defined and selected at instantiation:

```ruby
persona :assistant do ... end
persona :researcher do ... end

# at instantiation
agent = ResearchAgent.new(persona: :researcher)
```

### Tools

```ruby
tools :web_search, :send_email, :calendar
```

`tools` looks up each symbol in the Tool Registry. If a tool is not registered — because its spur gem isn't installed — it raises at class load time, not at runtime. You find out immediately, not when the agent tries to use it mid-task.

Individual tool configuration can override defaults:

```ruby
tools :web_search,
      :send_email  => { requires_confirmation: true },
      :calendar    => { timeout: 10 }
```

### Memory

```ruby
memory :short_term, window: 20
memory :long_term,  adapter: :postgres, embedding_model: :openai
memory :episodic
```

Each memory declaration registers a memory store. Multiple stores are active simultaneously and the agent consults them in order during context assembly. Short-term is always present — it cannot be disabled. Long-term and episodic are opt-in.

### Guardrails

```ruby
guardrails do
  injection_filter  :strict      # :strict, :moderate, :permissive
  pii_filter        :redact      # :redact, :block, :warn, :off
  max_tool_calls    5            # per turn limit
  max_turns         20           # per session limit
  denied_domains    ["competitor.com"]
  audit             :full        # :full, :minimal, :off
end
```

Guardrails are enforced by the Context Pipeline on every turn. They are not advisory — violations halt execution and raise typed errors that the developer can rescue and handle.

### Hooks

Hooks are the agent's event system. They are optional but powerful for observability, custom logging, and side effects.

```ruby
on_start        { |session| Rails.logger.info "Agent session #{session.id} started" }
on_turn_start   { |turn|    AuditLog.record_turn_start(turn) }
on_tool_call    { |tool, args, result| Metrics.increment("tool.#{tool}") }
on_turn_end     { |turn|    AuditLog.record_turn_end(turn) }
on_finish       { |session| notify_user(session.summary) }
on_error        { |error|   Sentry.capture(error) }
```

Hooks receive typed event objects, not raw hashes. They run synchronously by default. An async hook adapter will be available in a later release.

## The Lifecycle

An agent session has a defined lifecycle with clear state transitions. States are not just documentation — the agent enforces them and raises `InvalidStateError` if you try to call methods out of sequence.

```
UNINITIALIZED
     │
     ▼  .new(...)
  READY
     │
     ▼  .run(...) or .chat(...)
  RUNNING ◄──────────────────────┐
     │                           │
     ▼  (tool called)            │
  WAITING_FOR_TOOL               │
     │                           │
     ▼  (tool result returned)   │
  PROCESSING ────────────────────┘  (loop continues)
     │
     ▼  (stop condition met)
  FINISHING
     │
     ▼
  COMPLETE ──► or ──► ERROR
```

### Initialization

```ruby
def initialize(user: nil, session_id: nil, persona: :default, **opts)
  @user         = user
  @session      = Session.load_or_create(id: session_id, store: self.class.session_store)
  @persona      = resolve_persona(persona)
  @memory       = MemoryManager.new(self.class.memory_config)
  @tools        = ToolRunner.new(self.class.tool_config, guardrails: self.class.guardrail_config)
  @pipeline     = ContextPipeline.new(self.class.guardrail_config)
  @adapter      = AdapterRegistry.resolve(self.class.model_config)
  @audit        = AuditLog.new(@session)
  @state        = :ready

  run_hook(:on_start, @session)
end
```

No LLM calls happen during initialization. The agent is ready but idle. This makes instantiation fast, testable, and safe.

### Entry Points

Two public entry points exist depending on interaction pattern. Both stream by default — this is not optional. (ADR-001)

```ruby
# Single-shot: provide full input, stream output, session ends
agent.run("Research the top 5 competitors of Acme Corp") do |chunk|
  print chunk.text
end

# or as an enumerator
agent.run("Research the top 5 competitors of Acme Corp").each do |chunk|
  print chunk.text
end

# Conversational: multi-turn, session persists between calls, each turn streams
agent.chat("What is the weather in Greenville?") { |chunk| print chunk.text }
agent.chat("What about tomorrow?") { |chunk| print chunk.text }
```

Both entry points feed into the same internal call loop. `run` wraps it with a single-turn session. `chat` keeps the session open and accumulates memory across turns.

Chunks are typed objects, not raw strings:

```ruby
chunk.text        # the string content of this chunk
chunk.type        # :text | :tool_start | :tool_end | :done
chunk.turn        # which turn this chunk belongs to
chunk.session_id  # session identifier
```

The call loop buffers the stream internally. Tool calls cannot be dispatched until the full argument payload has arrived in the buffer — a partial tool call is never executed. Text chunks are yielded to the caller as they arrive. Memory updates and audit logging happen on stream completion, not mid-stream.

## The LLM Call Loop

This is the core engine. It runs until a stop condition is met. The loop is synchronous in v1 with async-ready interface boundaries (ADR-002). Responses stream to the caller as they arrive (ADR-001).

```
┌─────────────────────────────────────────────────────┐
│                   LLM Call Loop                      │
│                                                      │
│  1. ASSEMBLE CONTEXT                                 │
│     memory retrieval + current input + persona       │
│     → Context Pipeline (security, fencing)           │
│                                                      │
│  2. STREAM LLM RESPONSE                              │
│     adapter opens stream, yields chunks to caller    │
│     buffer accumulates internally                    │
│     tool call boundary detected in buffer?           │
│       → hold, do not yield, wait for full args       │
│                                                      │
│  3. PARSE BUFFERED RESPONSE                          │
│     text reply? → flush buffer, done for this turn   │
│     tool call?  → dispatch to tool runner            │
│                                                      │
│  4. EXECUTE TOOL (if requested)        [ASYNC-READY] │
│     permission check → execute → wrap result         │
│     → Content object with trust: :external           │
│     → yield tool_start / tool_end chunks to caller   │
│                                                      │
│  5. OBSERVE                                          │
│     tool result added to context                     │
│     memory updated (on stream completion only)       │
│     audit log updated                                │
│                                                      │
│  6. CHECK STOP CONDITIONS                            │
│     LLM said it's done?      → COMPLETE              │
│     max_tool_calls reached?  → COMPLETE              │
│     max_turns reached?       → COMPLETE              │
│     error raised?            → ERROR                 │
│     otherwise                → back to step 1        │
└─────────────────────────────────────────────────────┘
```

### Step 1 — Context Assembly

Context is assembled by the `ContextAssembler`, which pulls from all registered memory stores and combines them with the current input. The assembled context is an ordered array of `Content` objects — never raw strings.

```ruby
class ContextAssembler
  def assemble(input:, session:, memory:, persona:)
    [
      persona.render,                          # trust: :system
      memory.short_term.recent,               # trust: :user or :external (inherited)
      memory.long_term.retrieve(input),       # trust: varies by origin
      Gates::UserInput.wrap(input),           # trust: :user
    ].flatten.compact
  end
end
```

The assembled array is handed to the Context Pipeline, which applies injection scanning, permission enforcement, and data fencing before producing the final prompt string.

### Step 2 — Stream LLM Response

The adapter opens a streaming connection and yields chunks. The call loop has two responsibilities simultaneously: yielding text chunks to the caller, and buffering the full response to detect tool call boundaries. A tool call is never dispatched on a partial payload.

The adapter defines both `#call` (legacy, sync) and `#stream` (primary). v1 uses `#stream`. The `scheduler:` parameter is a sync no-op in v1 — this is the async seam. (ADR-002)

```ruby
class Adapter::Claude < Spurline::Adapter::Base
  # ASYNC-READY: scheduler param is the async entry point
  def stream(prompt:, tools:, config:, scheduler: Spurline::Scheduler::Sync, &chunk_handler)
    scheduler.run do
      client.messages.stream(
        model:      config.model_string,
        system:     prompt.system,
        messages:   prompt.messages,
        tools:      tools.to_schema,
        max_tokens: config.max_tokens
      ) do |raw_chunk|
        chunk = Chunk.parse(raw_chunk)
        chunk_handler.call(chunk)   # yield to caller
        buffer << chunk             # accumulate internally
      end
    end

    Response.parse(buffer)          # parse complete response after stream ends
  end
end
```

### Step 3 — Response Parsing

The response is parsed into a typed `Response` object. It is either a `TextResponse` (the agent is done for this turn) or a `ToolCallResponse` (the agent wants to invoke a tool).

```ruby
class Response
  def self.parse(raw)
    if raw.stop_reason == "tool_use"
      ToolCallResponse.new(raw)
    else
      TextResponse.new(raw)
    end
  end
end
```

This is a strict bifurcation. There is no ambiguous middle state.

### Step 4 — Tool Execution

`ToolCallResponse` carries a tool name and arguments. The `ToolRunner` handles execution. The `scheduler:` parameter is the async seam — sync no-op in v1. (ADR-002)

Tools are strictly leaf nodes — they cannot invoke other tools. Any attempt raises `NestedToolCallError` with a message pointing to the Skill layer. (ADR-003)

```ruby
class ToolRunner
  # ASYNC-READY: scheduler param is the async entry point
  def execute(tool_call, session:, scheduler: Spurline::Scheduler::Sync)
    tool = registry.fetch(tool_call.name)        # raises ToolNotFoundError if unknown

    permission_check!(tool, session)             # raises PermissionDeniedError if not permitted
    confirmation_check!(tool, session)           # yields :confirmation chunk to caller if required

    raw_result = scheduler.run { tool.call(**tool_call.arguments) }

    Gates::ToolResult.wrap(raw_result, tool_name: tool_call.name)
    # always returns Content with trust: :external
  end
end
```

The tool result is always wrapped in a `Content` object at the gate. It cannot enter the next loop iteration as a raw string. Two chunks are yielded to the caller around tool execution:

```ruby
# caller sees:
{ type: :tool_start, tool: :web_search, args: { query: "..." } }
# ... tool executes ...
{ type: :tool_end,   tool: :web_search, duration_ms: 340 }
```

### Step 5 — Observe

The tool result is appended to the session's working memory. The audit log records the tool name, arguments, result, and timestamp. The memory manager decides whether the context window needs trimming (summarization) before the next iteration.

### Step 6 — Stop Conditions

Stop conditions are checked in priority order:

```ruby
def stop?(response, session)
  return :agent_done      if response.is_a?(TextResponse)
  return :max_tool_calls  if session.tool_call_count >= guardrails.max_tool_calls
  return :max_turns       if session.turn_count >= guardrails.max_turns
  return :denied_domain   if violated_domain_policy?(session.last_tool_result)
  false
end
```

When a stop condition is met, the loop exits, `on_finish` hooks fire, memory is persisted, and the session is marked `:complete`.

## Error Handling

Errors inside the call loop are caught, classified, and routed:

```ruby
class AgentError            < StandardError; end
class TaintedContentError   < AgentError; end    # raw string in pipeline
class InjectionAttemptError < AgentError; end    # injection pattern detected
class PermissionDeniedError < AgentError; end    # tool not permitted
class PersonaFrozenError    < AgentError; end    # attempt to mutate system prompt
class InvalidStateError     < AgentError; end    # lifecycle state violation
class ToolNotFoundError     < AgentError; end    # tool called but not registered
class MaxToolCallsError     < AgentError; end    # guardrail limit hit
class NestedToolCallError   < AgentError; end    # tool attempted to call another tool (ADR-003)
```

All errors are logged to the audit trail before propagation. The `on_error` hook fires for every unhandled error. Developers can rescue specific error types:

```ruby
class MyAgent < ApplicationAgent
  on_error do |error|
    case error
    when Spurline::InjectionAttemptError
      notify_security_team(error)
    when Spurline::PermissionDeniedError
      log_and_gracefully_degrade(error)
    else
      raise
    end
  end
end
```

## Session Object

The `Session` object is the running record of everything that happened. The framework owns session persistence via an adapter interface — developers do not manage this themselves. (ADR-004)

```ruby
session.id                  # unique session identifier
session.agent_class         # which agent ran
session.user                # who invoked it
session.turns               # array of Turn objects
session.tool_calls          # flat list of every tool invoked (ADR-003: always flat)
session.tool_call_count     # integer count
session.turn_count          # integer count
session.state               # current lifecycle state
session.started_at          # Time
session.finished_at         # Time
session.duration            # seconds
session.audit_trail         # full structured log
session.summary             # auto-generated natural language summary
session.metadata            # free-form hash for application-defined data
```

Sessions are persisted automatically by the configured store. The default store is in-memory (development and testing). Postgres ships as the first production adapter.

```ruby
# Retrieve a past session by ID
session = Spurline::Session.find("uuid-here")
session.turns.each { |turn| puts turn.summary }

# Resume a conversational session
agent = MyAgent.new(session_id: "uuid-here")
agent.chat("Continue from where we left off")
```

Session resumption reloads the full turn history into short-term memory. The agent behaves as if the conversation never paused.

## Testing

Because the lifecycle is explicit and initialization makes no LLM calls, agents are straightforward to test. The framework ships test helpers that stub the adapter with streaming-compatible canned responses:

```ruby
# In a test
agent = ResearchAgent.new
agent.use_stub_adapter(responses: [
  stub_text("Here is what I found..."),
  stub_tool_call(:web_search, query: "Acme Corp competitors"),
  stub_text("Based on my research...")
])

chunks = []
agent.run("Research our competitors") { |chunk| chunks << chunk }

assert_equal :complete,       agent.session.state
assert_equal 1,               agent.session.tool_call_count
assert_includes chunks.map(&:text).join, "Based on my research"

# streaming chunk types are testable too
assert chunks.any? { |c| c.type == :tool_start }
assert chunks.any? { |c| c.type == :tool_end }
```

Stubs play back canned responses as chunked streams. The full pipeline — injection scanning, permission checks, memory management — still runs against the stubbed responses. You're testing real agent behavior, just without live LLM calls or network I/O.

## Architectural Decisions

All open questions from initial design have been resolved. See the ADR document for full rationale.

| Decision | Resolution |
|----------|-----------|
| Streaming | Streaming first — always. `#run` and `#chat` yield chunks. No non-streaming mode. (ADR-001) |
| Async | Sync-first with async-ready interfaces. `# ASYNC-READY:` marks seam points. (ADR-002) |
| Nested tools | Leaf nodes only in v1. Tools cannot invoke tools. Composition belongs in Skills. On roadmap. (ADR-003) |
| Session storage | Framework-owned with adapter interface. Memory default, Postgres first production adapter. (ADR-004) |
